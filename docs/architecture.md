## Components

### `sinhala_ime::Mapping`

Parses `phonetic-default.json` into two tries and a handful of codepoints.

- **Consonant trie**: keys are `std::u32string` (typically length 1 or 2).
  Single-key consonants with a Shift variant occupy `{c}` and
  `{shift_key(c)}`. Multi-key consonants (e.g. `"th"`, `"^g"`) occupy
  multi-character keys. Aspirated digraphs with a capital first letter
  (e.g. `"Th"`) use `{shift_key('t'), 'h'}`.
- **Vowel trie**: same shape, payload is `{standalone, matra}` codepoints.
- **Special codepoints**: `virama`, `anusvara`, `visarga`, `zwj`.
- **Rule keys**: `terminator_key` (`x`), `conjunct_marker` (`X`),
  `anusvara_key` (`q`), `visarga_key` (`Q`), plus `unmapped_passthrough`.

Throws `std::runtime_error` on parse, schema, or file-IO errors.

### `sinhala_ime::Trie<T>`

Header-only generic prefix trie keyed on `char32_t` sequences. Supports
`insert`, exact `find`, `longest_prefix_value`, `is_strict_prefix`, and
`is_prefix`. Allocation occurs only at `insert`; all lookup paths are
allocation-free.

The **Shift-bit convention** uses bit 31 of a `char32_t` to mark
"Shift-modified ASCII." Real Unicode stops at U+10FFFF (21 bits) so the
high bit never collides with a legal codepoint.

### `sinhala_ime::Composer`

The state machine, coded directly from the transition table in
`docs/phonetic-mapping-design.md`. Holds `CompositionState`, a buffered
consonant codepoint, and a `std::u32string` prefix buffer (reserved once at
construction).

Five states: `Idle`, `PartialConsonant`, `ConsonantPending`, `PartialVowel`,
`PartialVowelStandalone`. Every transition is either (a) committed to
output, (b) a buffer update, or (c) a recursive re-feed of the same input
against a new state — the table drives this explicitly to keep the code
readable.

### `sinhala_ime::Engine`

Thin façade over `Composer`. Factories accept either a JSON file path or a
JSON string. Never throws across the API boundary; on any internal error it
returns a `FeedResult` with `consumed=false` and empty strings.

### Python reference (`prototype/transliterate.py`)

Authoritative source of truth for the state-machine semantics. Every
change to the state machine must be made in Python first, then
`tools/generate_golden.py` regenerates the cross-validation golden file,
and the C++ `Composer` is updated to match byte-for-byte.

## Data flow (per keystroke)

1. Host TSF DLL (Phase 3) receives a `WM_KEYDOWN`-like event.
2. It translates to a `sinhala_ime::KeyEvent` (codepoint + modifiers).
3. `Engine::feed` -> `Composer::feed`.
4. `Composer` does trie lookups, updates state, potentially emits UTF-8.
5. Result returns: `{committed, composition, consumed}`.
6. TSF DLL inserts `committed` into the document and updates the
   composition-display with `composition`.

## Design choices & trade-offs

### Why `char32_t` and the Shift-bit convention?

Keeps single-key and multi-key + shifted entries in one trie with one type.
Alternatives considered:
- Two separate tries (extra structural complexity, identical semantics).
- Encoding Shift in a pair struct (larger keys, slower hashing).
- Using `std::string_view` ASCII only and then post-processing shift
  (breaks the "longest-match wins" rule at the boundary).

### Why an explicit state machine vs. a big lookup?

The "longest-match wins" requirement means the engine must buffer ambiguous
prefixes until disambiguation. A stateless lookup-table approach would need
unbounded re-feeds. The state machine is small enough to fit in one
transition table and keeps every per-keystroke cost O(1) in expectation
(trie hash-map lookups).

### Why is Python the source of truth?

- Faster iteration on the state machine while it was being designed.
- Easier to reason about invariants in untyped prose + short Python.
- The cross-validation gate (golden file) enforces C++↔Python parity
  mechanically — if the C++ engine drifts, CI fails on a diff.

### Why no auto-nasalization of `nk`/`ng`/`mp`/`mb`?

Phonological rules would need context across multiple symbols, complicating
the state machine and violating the "one character in, one transition out"
model. Anusvara via an explicit `q` key is a Phase-1 choice documented in
the design doc; auto-nasalization is listed as a Phase-2.5 enhancement.

### Why no exceptions across the API boundary?

TSF COM interop later — exceptions do not cross COM boundaries. Internal
logic may throw; `Engine::feed`/`commit` catch and degrade gracefully.

## Performance

Hot-path `feed()`:
- At most two trie hash-map lookups per key (exact + prefix).
- No allocation on the hot path **except** to grow the returned
  `std::string`s (SSO typically covers the common case). Composition
  buffer is reserved once at construction.

Target: < 100 µs per keystroke on a modern CPU. Not enforced by CI;
benchmarks are a future task.

## Extension points

- **Alternative keymaps**: load a different `*.json` file.
- **Candidate windows / suggestions**: layer on top of `Engine::feed`;
  the engine is purely mechanical, no learning.
- **Custom rule keys**: swap `terminator_key`, `anusvara_key`, etc. via
  the JSON `rules` block.
